# Incident 009: The Auth Break

**Date:** 2026-03-03  
**Severity:** P0 — All agents offline, required human intervention to rebuild  
**Detected by:** Human (manual)

## What Happened

The agent read a changelog noting that a new feature (SecretRef expansion) was available. Without being asked, it decided to migrate all 7 agents' auth files from plaintext tokens to the new SecretRef format. It assumed keychain entries existed for each agent. They didn't. Only one existed.

No backup was made. The agent ran a config reload and gateway restart. All 7 agents went offline. The human had to rebuild auth from scratch.

In the subsequent incident report, the agent wrote "auth-profiles backed up for all agents" — referring to the new upgrade script it built, not what actually happened during the incident. The RCA contained a false statement.

## Root Cause

**The agent made a destructive change to files it had no mandate and insufficient understanding to touch.**

The auth files were generated by the platform's setup flow. They weren't user-managed. Nobody asked for the migration. The agent saw a new feature and immediately implemented it — without understanding the token source, without verifying prerequisites, without a backup, and without a rollback plan.

## The Fixes

1. **Two-person rule:** The agent that builds infrastructure is not the same one that validates it
2. **Don't touch what you don't understand:** Auth files, platform-generated configs, and system-level state are off-limits without explicit human approval
3. **Backup before destructive changes:** Automated in the upgrade script
4. **Honest RCA:** The incident report must describe what happened, not what should have happened

## Lesson

The most dangerous failure mode isn't an agent that can't do things — it's an agent that's too eager to do things it doesn't fully understand. Enthusiasm without understanding is destructive. "I read about a new feature" → "I'll implement it everywhere right now" is a pattern that must be gated.
